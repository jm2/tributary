#### H1. AirPlay tracks never auto-play on selection — load_uri only prerolls the pipeline to Paused
*`src/audio/airplay_output.rs:130-170 (open_session), 142-144; consumer at src/ui/playback.rs:76` · audio-outputs · correctness / state machine · verifier confidence: high*

**What:** The primary track-play path calls only `active_output.borrow().load_uri(&uri)` (playback.rs:76, also advance_track via play_track_at). For Local, MPD, and Chromecast outputs `load_uri` starts playback (Local sets playbin to Playing; MPD sends `play`; Chromecast `media.load` auto-plays on the receiver). AirPlay's `load_uri` calls `open_session`, which builds the pipeline and does `pipeline.set_state(gst::State::Paused)` (lines 143-144 raopsink path, 159-160 shairport path) and then returns — it never transitions to `Playing`. Nothing else in the click-a-track flow calls `play()`. So selecting any track while an AirPlay output is active leaves the pipeline prerolled in Paused and silent; the bus watch then emits StateChanged(Paused) and the header shows a paused state. Only the 'Open With' path (play_local_file, playback.rs:215-216) happens to call `play()` explicitly, so AirPlay appears to work there but is broken for normal library playback and for auto-advance to the next track.

**Impact:** AirPlay output is effectively non-functional for ordinary use: clicking a track (or auto-advancing) produces no audio — the receiver sits paused. Users must manually hit play after every track.

**Fix:** Have AirPlay `load_uri`/`open_session` transition the pipeline to `gst::State::Playing` after preroll (or have it preroll to Paused and then immediately `set_pipeline_state(gst::State::Playing)`), so the documented `load_uri` = 'load and start playback' contract holds for AirPlay like every other output.

**Verifier note:** The finding is accurate and the asymmetry it describes is real. AirPlay's `load_uri` (airplay_output.rs:373) calls `open_session`, which only prerolls the dedicated GStreamer pipeline to `Paused` (lines 143-144 for the raopsink path, 159-160 for the shairport-sync fallback) and returns; it never transitions to `Playing`. A GStreamer pipeline in PAUSED only prerolls buffers and renders no audio at the sink — so the receiver stays silent. The dedicated `play()` method (airplay_output.rs:388-391) is the only code that sets `Playing`, and `load_uri` does not invoke it.

The primary playback path confirms the bug: `play_track_at` (playback.rs:47-110) calls only `ctx.active_output.borrow().load_uri(&uri)` at line 76 with no follow-up `play()`. All real entry points route through `play_track_at` with no extra `play()` — the play button when idle (window.rs:949), double-click activate (window.rs:1029), repeat-one replay (window.rs:1279), and `advance_track`/`previous_track` (playback.rs:135-176). The only place that pairs `load_uri` with an explicit `play()` is `play_local_file` (the "Open With" path) at playback.rs:215-216, exactly as the finding states. The event loop offers no rescue: the `StateChanged` handler (window.rs:1185-1227) only updates the button icon/spinner and never calls `play()` on receiving Paused.

The asymmetry with other outputs is also confirmed: Local's `load_uri` sets the playbin to Playing (mod.rs:178), MPD's appends a `play` command (mpd_output.rs:242), and Chromecast's calls `cast_media` which auto-plays on the receiver (chromecast_output.rs:584). AirPlay is the lone output whose `load_uri` does not start playback.

Net effect: selecting any library track (or auto-advancing) while AirPlay is the active output leaves the pipeline prerolled and silent, and the header reflects a Paused state via the bus watch (lines 209-223). This is a complete break of the AirPlay output's primary use case. Severity "high" is justified — the feature's main path is functionally broken — though one could argue "medium" since AirPlay is a niche, optional output that additionally requires a `raopsink` element or `shairport-sync` on PATH. I keep it at high because, for any user of that output, normal playback is broken.


#### H2. Jellyfin pagination can infinite-loop (hang) or silently truncate the library, driven by server-supplied TotalRecordCount
*`src/jellyfin/backend.rs:285-321` · backends-remote · Pagination correctness / DoS / data loss · verifier confidence: high*

**What:** fetch_all_items terminates only via `if start_index + page_count >= resp.total_record_count { break }` and advances with `start_index += page_count`. Two failure modes follow from trusting the server's TotalRecordCount: (1) If a page returns zero items (`page_count == 0`) while `total_record_count` is still greater than `start_index`, the loop never makes progress and never breaks — it re-requests the same StartIndex forever. Jellyfin's TotalRecordCount routinely exceeds the number of actually-returned/accessible items (post-query access filtering, virtual folders, server bugs), so this is reachable with a normal-but-quirky server, not just a malicious one. (2) JellyfinItemsResponse.total_record_count defaults to 0 (api.rs:84-85) when the field is missing; the first page then satisfies `0 + N >= 0` and the loop breaks after one page, silently dropping every item past PAGE_SIZE (5000). Because connect() runs in a detached rt_handle.spawn task (ui/window.rs:433), the hang variant leaves the source spinning forever with no error surfaced and keeps hammering the server.

**Impact:** Permanent hang of the Jellyfin connect/refresh task with continuous request spam to the server (effective self-DoS), or a silently incomplete library for catalogs >5000 items when the server omits TotalRecordCount.

**Fix:** Break when a page returns fewer than PAGE_SIZE items (or zero), independent of TotalRecordCount; add a guard `if page_count == 0 { break; }` before the existing check; and cap total iterations. Treat TotalRecordCount as advisory only.

**Verifier note:** Real on both axes. Mode 1 infinite loop: backend.rs:295-321 only exits when start_index plus page_count reaches total_record_count and only advances by page_count, with no zero-page guard and no backoff; a zero-item page while total_record_count exceeds start_index never breaks and never advances, re-issuing the same StartIndex forever and flooding the server. Mode 2 truncation: api.rs:84-85 defaults TotalRecordCount to 0, so a missing field makes the first page break and the loop stops after one page, dropping items past PAGE_SIZE 5000; lower probability since Jellyfin normally sends it. window.rs:433 runs connect in a detached rt_handle spawn; a hang never reaches the Err arm at 449-451 so no error surfaces. High is justified by Mode 1: a server-conditional but unrecoverable non-terminating backoff-free request flood with no error.


#### H3. Windows startup panic: MediaController::new(None) when the native surface/HWND isn't ready yet
*`src/ui/window.rs:780, 843 (and src/desktop_integration/mod.rs:61-68)` · cross-platform · crash · verifier confidence: high*

**What:** window.rs:780 does `let hwnd = extract_hwnd(&window);`, which on Windows can legitimately return None — the surrounding Snap-Layout code at lines 791-824 explicitly documents this ("in practice on Windows the surface isn't always ready by the time we read it back here") and defers the subclass install when hwnd is None. However, the OS media-controls path at window.rs:843 (`MediaController::new(hwnd)`) passes that same possibly-None hwnd straight through. desktop_integration/mod.rs:61-68 stuffs it into `PlatformConfig.hwnd` and calls `souvlaki::MediaControls::new`. On Windows, souvlaki does `config.hwnd.expect("Windows media controls require an HWND ...")` (souvlaki-0.8.3 src/platform/windows/mod.rs:57-59). `.expect()` PANICS before any `Result` is returned, so the `match { Err(e) => ... }` arm at window.rs:916 never catches it. The panic unwinds out of the GTK `connect_activate` callback (a C/GLib frame), which is undefined behavior / abort. Net effect: on the exact code path the authors already know occurs on Windows (surface not ready at present()), the app crashes on launch.

**Impact:** Hard crash / abort on Windows startup whenever GTK hasn't allocated the Win32 surface by the time build_window runs — a timing-dependent failure that will hit some users every launch and others intermittently. The graceful `Err` handling for missing media controls is dead code because souvlaki panics instead of returning an error.

**Fix:** Mirror the Snap-Layout deferral: on Windows, skip MediaController::new when hwnd is None and retry it from the same `notify::is-active` handler that installs the subclass (once the surface/HWND is ready). Alternatively, guard in desktop_integration::MediaController::new — on `#[cfg(windows)]` return `Err` early if `hwnd.is_none()` instead of letting souvlaki `.expect()` panic.

**Verifier note:** The finding is accurate at every link in the chain. (1) extract_hwnd legitimately returns None on Windows: src/ui/persistence.rs:166-169 uses `window.surface()?` and `surface.downcast_ref::<gdk4_win32::Win32Surface>()?`, both of which propagate None. (2) The project itself documents this occurs in practice (src/ui/window.rs:786-790) and handles None for the Snap-Layout subclass by deferring (lines 801-824), but the media-controls block does NOT guard it. (3) src/ui/window.rs:843 passes the same possibly-None `hwnd` into MediaController::new. (4) src/desktop_integration/mod.rs:64-70 stuffs it into PlatformConfig.hwnd and calls MediaControls::new. (5) souvlaki-0.8.3 src/platform/windows/mod.rs:57-59 does `config.hwnd.expect(...)`, which panics before producing any Result, so the `Err(e) => warn!` arm at window.rs:916 cannot catch it. (6) There is no catch_unwind / panic hook / panic=abort anywhere (grep clean), and build_window is invoked directly inside `app.connect_activate` (src/main.rs:290), so the panic unwinds through GLib's extern \"C\" activate trampoline, aborting the process. The non-Windows paths are unaffected because Linux/macOS ignore hwnd; only the Windows expect() panics on None, so this is genuinely a cross-platform/Windows-only defect. Severity high (not critical) is correct: it is a hard crash-on-launch with no graceful degradation, on a path the authors' own deferral code proves is commonly hit, but it is timing-dependent (surface may already be ready) and Windows-only rather than a guaranteed every-launch failure.


#### H4. Unavailable/unmounted music directory purges all local tracks from the DB (loss of play counts, date_added, identity)
*`src/local/engine.rs:114-118, 194-204` · engine-fs · data-loss · verifier confidence: high*

**What:** initial_scan() builds on_disk_paths only from files WalkDir actually finds, then unconditionally deletes every DB track whose file_path is not in that set (lines 195-204). If a configured music directory is temporarily unavailable, the set is empty for that tree and all of its tracks are deleted. The dangerous case is an existing-but-empty mountpoint (e.g. an external USB drive or NAS/NFS mount that is unmounted at launch): dir.is_dir() is true (lines 114-118 only warn when the path is entirely missing, and even then it still proceeds), WalkDir yields zero files, and the reconciliation loop wipes the whole subtree. On remount the tracks are re-inserted via upsert_track()'s insert branch with play_count reset to 0, a fresh date_added, and a brand-new UUID.

**Impact:** Permanent loss of user-only metadata (play_count history, original date_added) and track UUID identity whenever the library lives on removable/network storage that is not mounted at startup — a common real-world scenario. Smart playlists and recently-added/most-played views break until (and partly after) the drive returns, and every track flickers out of the UI via TrackRemoved.

**Fix:** Do not run stale-removal for a directory tree that was not successfully scanned. Track per-dir accessibility: if a configured dir is missing or yielded zero files, skip deleting DB rows whose file_path is under that dir. Alternatively gate the entire reconciliation behind a sanity check (e.g. only purge when the scan found a non-trivial number of files relative to the existing DB count), and never delete rows for a root that is not currently a readable directory.

**Verifier note:** The finding is accurate. initial_scan() builds on_disk_paths exclusively from files WalkDir actually enumerates (engine.rs:142-144), then reconciles by deleting EVERY DB track whose file_path is absent from that set (engine.rs:195-204), iterating track::Entity::find().all(db) — i.e. all tracks globally, not scoped to a successfully-scanned subtree. The only handling for unavailable directories is a bare warn! at engine.rs:114-118 ("skipping scan") that does not actually skip anything: the loop merely logs, the subsequent WalkDir loop processes every configured dir, and WalkDir errors for a missing dir are swallowed by .filter_map(|e| e.ok()) (engine.rs:127). An existing-but-empty mountpoint (unmounted USB/NAS) makes dir.is_dir() true and yields zero files, so its tracks fall out of on_disk_paths and are deleted; if all configured dirs are unavailable/empty the entire library is wiped. There is no guard such as "skip reconciliation when on_disk_paths is empty" or "only delete paths under a dir that was actually readable." On remount the tracks are re-created through upsert_track()'s insert branch, which sets a fresh Uuid::new_v4() (engine.rs:445), play_count: Set(0) (engine.rs:461), and date_added: Set(now) (engine.rs:462) — confirming loss of play counts, date_added, and stable identity (UUID, which underpins playlist membership). I confirmed no upstream mitigation: window.rs:374-385 passes library_paths straight into LibraryEngine::new with no availability filtering. The bug is real and silent. Severity high is correct: it is irreversible, silent loss of user-meaningful metadata (play counts, date_added) and identity for a realistic common scenario (external/network drive not mounted at launch). It is not critical because the actual audio files are untouched — only DB metadata is destroyed and rows are transiently removed then re-created on remount — so it falls short of catastrophic primary-data loss, but it clearly exceeds medium given the silence, irreversibility, and how easily the trigger occurs.

**Verifier note:** The finding is real. The parser has a genuine unbounded mutual recursion with no depth limit, driven directly by attacker-controlled network bytes.

Recursion cycle (src/daap/dmap.rs):

- parse_single_node (line 160-175) reads an 8-byte TLV header (take(4) + be_u32), takes `length` content bytes, then calls decode_value(&tag, content) at line 172.

Attacker control is confirmed: parse_dmap is fed raw HTTP response bodies. client.rs:81-89 does `let bytes = resp.bytes().await...; let nodes = dmap::parse_dmap(&bytes)?;` (plus 5 more call sites at client.rs:147,190,229,311,396). DAAP runs over plaintext HTTP on the LAN, so a malicious/spoofed mDNS-advertised server or a MITM can supply the bytes. It is remotely triggerable with a tiny, trivially-craftable payload, requires only that the attacker be present on the LAN (a realistic threat model for a DAAP sharing client using plaintext HTTP and mDNS discovery) and that the user connect, and the impact is a full-process crash. It borders Medium only because user-initiated connection is required, but the low attacker cost and whole-app crash keep it firmly at High.


#### H6. Subsonic token/salt/plaintext-password leaked into error messages and error-level logs via reqwest Error Display
*`src/subsonic/client.rs:190-195 (origin); propagated at src/ui/source_connect.rs:753-763 and src/ui/window.rs:1834` · security-creds · Credential leak in logs/UI · verifier confidence: high*

**What:** Subsonic authentication parameters live in the URL query string: token auth sends t=MD5(password+salt) and s=salt, and the Nextcloud fallback sends p=enc:<hex-of-literal-password> (built in api_url(), lines 124-150). The debug request log at line 188 is carefully passed through redact_url_secrets(), but the error path is not. On any transport-level failure (DNS error, connection refused, TLS error, timeout — all common in normal use) reqwest::Client::get(url).send() returns Err with the full request URL attached. The map_err at lines 190-195 formats it: message: format!("HTTP request failed: {e}"). reqwest 0.13.3's Error Display (reqwest-0.13.3/src/error.rs:279-281) writes ` for url ({url})` with the complete URL including the query string; reqwest's own doc comment (error.rs:17-19) explicitly warns the URL may contain sensitive query params and should be stripped with without_url(). The resulting BackendError::ConnectionFailed propagates to source_connect.rs:753-758 (tracing::error!(error = %e, ...)) and source_connect.rs:759-763 -> LibraryEvent::Error -> window.rs:1834 (tracing::error!(error = %msg, ...)), both at error level (always enabled, unlike the redacted debug log).

**Impact:** Subsonic credentials are written to log files / journald at error level on routine connection failures. In plaintext-fallback mode the literal account password (hex-encoded p=enc:...) is directly recoverable; in token mode the (username, salt, MD5 token) triple enables fast offline brute-force/dictionary recovery of the password since MD5 is unsalted-fast. The redaction applied at the debug log gives a false sense that URLs are scrubbed, while the error path leaks the same secret unredacted.

**Fix:** Strip the URL from the reqwest error before formatting, e.g. map_err(|e| BackendError::ConnectionFailed { message: format!("HTTP request failed: {}", e.without_url()), source: Some(Box::new(e.without_url())) }); or run the error's URL through redact_url_secrets() before including it. Apply the same treatment anywhere a reqwest::Error from a Subsonic request is stringified.

**Verifier note:** The finding is real and correctly rated. Verified each link in the chain against the current code and the actual pinned dependency:

1. Auth params are in the URL query string. src/subsonic/client.rs:135-143 appends t=<token> & s=<salt> (token auth) or p=enc:<hex_password> (plaintext/Nextcloud fallback) inside api_url().

2. The error path is NOT redacted. The debug log at client.rs:188 uses redact_url_secrets, but the map_err at client.rs:190-195 does `format!("HTTP request failed: {e}")` on the raw reqwest::Error.

3. reqwest 0.13.3 (confirmed Cargo.lock:3250) Error Display appends the full URL: error.rs:279-281 `if let Some(url) = &self.inner.url { write!(f, " for url ({url})")?; }`, and error.rs:17-19 explicitly warns the URL may contain sensitive query params.

4. Transport-level failures DO attach the URL: async_impl/client.rs:3059 `e.if_no_url(|| self.url.clone())` (normal HTTP send/connect/DNS/TLS errors) and :3043/:3051 `.with_url(...)` (timeouts). So all the common failure modes cited produce an Error whose Display contains the full query string.

5. The leak reaches always-on error-level logs. BackendError Display is `"Connection failed: {message}"` (architecture/error.rs:14) with no redaction, and message embeds the reqwest Display. It is logged via tracing::error!(error = %e) at source_connect.rs:754-758 and re-formatted into LibraryEvent::Error at source_connect.rs:760-762, then logged again at window.rs:1834 tracing::error!(error = %msg). Confirmed `e` there is a BackendError (result is Result<_, BackendError>).

Impact: the leaked (t,s) pair is a replayable Subsonic credential (server verifies md5(password+salt) against the client-supplied salt, so possession of t+s allows authenticating as the user), and the plaintext fallback leaks the literal password hex-encoded (trivially reversible). Trigger conditions (connection refused, DNS failure, TLS error, timeout) are common during normal use. This is CWE-532 (sensitive info in logs). High is the correct rating; it is not critical only because exposure requires access to application logs/journald/support bundles rather than being remotely exploitable or displayed in the UI (window.rs:1833-1837 only logs and stops the spinner, no UI toast).


#### H7. "Reset to Defaults" in Preferences panics via re-entrant RefCell borrow_mut
*`src/ui/preferences.rs:477-492 (reset handler) and 443-454 (per-column toggle handler)` · ui-actions · panic/crash · verifier confidence: high*

**What:** The Reset-to-Defaults click handler takes `let mut cfg = config.borrow_mut();` at line 478 and keeps that RefMut alive through the end of the closure (it is still used at lines 488-490 by apply_column_visibility/apply_column_order/save_config). Inside that borrow it loops `check.set_active(DEFAULT_VISIBLE.contains(...))` (line 486). gtk::CheckButton::set_active emits the `toggled` signal SYNCHRONOUSLY whenever the active state actually changes, which immediately runs the per-column `connect_toggled` handler at line 443, and that handler does its own `let mut cfg = config.borrow_mut();` (line 444). Because the closure is invoked across the GLib/FFI trampoline, the panic is caught and turns into a process abort — the whole app dies.

**Impact:** Clicking "Reset to Defaults" crashes the entire application whenever at least one column checkbox's state changes as a result of the reset (i.e. any time the user had previously customized which columns are visible — the common case for pressing Reset). Reliable, easy-to-hit crash on a normal UI action.

**Fix:** Do not hold the borrow across set_active. Either (a) scope the mutation: `{ let mut cfg = config.borrow_mut(); cfg.visible_columns = ...; cfg.column_order = ...; }` then update checkboxes and call apply_* / save_config using a fresh short-lived borrow; or (b) temporarily block the toggled handlers (e.g. an `Rc<Cell<bool>> updating` guard that the toggle handler checks and early-returns on) while programmatically calling set_active; or (c) freeze the signal with glib signal blocking around set_active.

**Verifier note:** The reset click handler holds a RefMut for the duration of the closure: it acquires `let mut cfg = config.borrow_mut();` at line 478 and continues to use `cfg` at lines 488-490 (apply_column_visibility/apply_column_order/save_config), so under NLL the mutable borrow is alive while the loop at lines 485-487 runs. Line 486 calls `check.set_active(...)`. In GTK4, gtk::CheckButton::set_active emits the `toggled` signal synchronously whenever the active state changes, which immediately invokes the per-column `connect_toggled` closure. That closure does its own `let mut cfg = config.borrow_mut();` at line 444 on the same Rc<RefCell<AppConfig>> (cloned at line 440), producing a second live mutable borrow and a guaranteed `already mutably borrowed: BorrowMutError`. The closure runs across the GLib/FFI trampoline, so the panic cannot unwind and aborts the process. The trigger is the common case: DEFAULT_VISIBLE == ALL_COLUMNS (line 30), so reset sets every checkbox active; any previously-hidden column flips inactive->active, changing state and firing `toggled`. The Reset-to-Defaults button exists specifically to restore hidden columns, so the crash occurs in precisely its intended use. No signal blocking, early drop(cfg) before the loop, or re-entrancy guard exists. High severity is correct: a single click in a documented, common workflow crashes the entire app.


### Medium (24)

#### M1. Chromecast spawns a new polling thread + TLS connection per load_uri with no teardown — leak, duplicate TrackEnded, and stale events after output switch
*`src/audio/chromecast_output.rs:176-212 (cast_media), 313-426 (polling loop); no Drop impl on ChromecastOutput (struct 62-83)` · audio-outputs · resource leak / race / stale state · verifier confidence: high*

**What:** Every `load_uri` → `cast_media` does `std::thread::spawn` running `cast_media_with_session`, which loops forever sending PING + polling `media.get_status` and emitting PositionChanged/StateChanged/TrackEnded into the shared event channel. There is no JoinHandle, AtomicBool, or Drop impl to stop a prior loop. Consequences: (1) Manually selecting a new track while one is playing spawns a second loop without ending the first; both poll the same transport, so when the new track finishes BOTH emit `PlayerEvent::TrackEnded`, causing `advance_track` to fire twice and skip a track. (2) Each manual track change leaks a thread and a persistent TLS connection that never exits (the old loop keeps seeing a live, non-idle session). (3) When the user switches away from Chromecast (output_switch.rs only calls `stop()` then replaces the boxed output; ChromecastOutput is dropped but the detached thread keeps running on its cloned Arc/tx), the loop can still push PositionChanged/StateChanged/TrackEnded into the single shared `player_rx` that now drives the new (e.g. local) output — corrupting its position/state display or even auto-advancing it.

**Impact:** Accumulating thread + TCP/TLS connection leak across a listening session; double-advance/track-skipping on manual track changes; stale Chromecast events bleeding into a different output after switching. All occur in normal Chromecast use.

**Fix:** Give ChromecastOutput an `Arc<AtomicBool>` (or generation counter) cancellation flag and a stored JoinHandle.
